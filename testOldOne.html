CG Definition

CG is a subfield of computer science dealing with the creation, processing and synthesis of visual content.

Rasterbild

- wurde vom Computer erzeugt
- ist das zu zeigende Bild → besteht aus einer Menge von Pixeln (= Atomen unseres Bildes)

Pixel

  • Definition: Pixel ist ein Tripel aus R, G, B Werten, zumeist jeweils eine Integerzahl im Bereich 0 - 255 (IN HEX 0-FF)
  • Datentyp: integer 0 - 255 (oder HEX 0 - FF)
  • Pixel Koordinaten:
    • Darstellung auf einem Rasterdisplay mit Funktion:
      setPixel(int x, int y, color color) //with color as RGB
      Jede Pixel auf dem Bildschirm kann und muss über seine Integer Koordinaten angesprochen werden.
    • Integerkoordinaten für die finale Ausgabe auf dem Bildschirm, NICHT für Verarbeitungsschritte
      Gründe:
      • Verschiedene Displayauflösungen
      • Dynamische Elemente
      • Mischen von Inhalten
      • 3D Darstellung
      → brauchen Abstraktion von den Bildschirmkoordinaten
  • Floating Point Zahlen (Alternative)
    2 Varianten:
    • double: 64-bit groß, 17 Stellen
    • float: 32-bit groß, 7 Stellen

Rasterization

  • Definition: Der Vorgang, von einer reinen Gedankenwelt von imaginären float Koordinaten wieder zurück in di darstellbare Welt der Pixel zu finden.
    (dies ist in modernen Geräten Aufgabe der Hardware)
  • Die Aufgabe des Pixelerzeugens übernimmt entweder:
    • die Grafikkarte (dezidierte Grafik) oder
    • ein spezieller Teil der CPU (integrierte Grafik)
    → In manchen Notebooks kann man zwischen beiden Varianten umschalten
  • Grafik-Power
    • Aufwand sehr hoch → Leistung von CPU übersteigt um einen Faktor 50-100
    • gleichzeitig gut parallelisierbar
  • GPU (Graphics Processing Unit): hat tausende Kerne → ernorme Leistungsfähigkeit
  • SIMD ("single instruction, multiple data")
    Besonders gut sind GPU zu nutzen, wenn wir ein und dieselbe Aufgabe mit verschiedenen Daten durchführen müssen.

Vertex - Primitive - Fragment - Pixel

  • Vertex: Sammlung von Attributen eines Punktes
  • Primitive: Einfaches Objekt durch Verticles (not grid-aligned)
  • Fragment: Teil eines Primitives in der Größe eines Bildschirmpixels (grid-aligned)
  • Pixel: Ding, das leuchtet

Shader

  • Definition
    Es gibt zwei Stellen, an denen wir den Lauf der Geschichte programmatisch beeinflussen können, diese Programme nennt man Shaders.
  • 2 Programme/Shaders
    Vertex Shader Fragment Shader
    • bekommt Daten direkt von CPU
    • Sie werden als Attributes zur Verfügung gestellt
    • Er führt nochmal letzte Berechnungen auf den Daten durch,
    • z.B. ein Dreieck in der Welt zu verschieben
    • Für jede Primitive werden also "pixelgroße" Fragmente herausgebrochen.
    • Für jedes dieser Fragmente wird nun der Fragment-Shader aufgerufen.
    • Er kann auf bestimmte "interpolierte" Attribute der die Primitive bestimmenden Vertices zugreifen.
    einmal aufgerufen (für jede Vertex) mehrmals aufgerufen (frei programmierbar für jede Fragmente)
    minimales Ergebnis: die Position in NDC (Normalized Device Coordinates) Ergebnis/Output: Farbe für Pixel

  • NDC (Normalized Device Coordinates)

    • Vertex Koordinaten werden nur in NDC berücksichtigt.
    • gelter Bereich: [-1.0, 1.0]
  • Datenfluss: OpenGL Rasterization Pipeline
    1. kriegt Daten von der CPU (Attribute Sets) an den Vertex Shader
    2. Vertex Shader bestimmt die NDC-Position und varying Attributes
    3. Rasterer schneidet alles weg, was nicht in [-1; 1] ist und...
    4. ...bestimmt Fragments in Pixelgröße für alls Primitive
    5. hierbei werden den Fragment varying Attributes gegeben
    6. Fragment Shader berechnet die potentielle Pixelfarbe...
    7. ...diese werden noch gemischt/verworfen.
  • WebGL Code
    • HTML Rohbau:
      var canvas = document.getElementById("mycanvas");
      var gl = canvas.getContext("webgl");
    • Funktion: säubern den Canvas mit bestimmter Farbe
      gl.clearColor(R, G, B, A); //clear canvas with RGB color and alpha (R, G, B and A values between 0-1)

Allgemein

  • Das Problem
    Ein festgelegtes Koordinatensystem ist NICHT GUT
    → Grund: man kann nicht auf Resourcen zurückgreifen:
    • die nicht extra für Device oder OPenGL
    • die mehrfach verwendet (verschiedene Positionen, Skalierung, Rotationen)
    • die interaktiv verändert
  • Lösung
    • jedes Objekt kriegt sein eigenes Koordinatensystem anstatt eines festgelegten System
    • dazu braucht man:
      • Model Koordinatensystem: ein natives Koordinatensystem
      • Ein Beschreibung: wie man von diesem in OpenGL Koordinatensystem kommt
      • Eine Koordinatentransformation

Koordinatensystem

  • Koordinatenursprung
    • ein eindeutig festgelegter Punkt
      • Notation der Punke: mit einem Tupel `(x, y,...)` notiert
      • Die Anzahl an Werten des Tupels = Dimension der Welt
    • Eine Menge von Achsen (2D mit `x, y`; 3D mit `x, y, z`)
      • Notation der Vektoren: wie Punkte `(x, y,...)`
      • Die Anzahl an Achsen = Dimension der Punktekoordinaten
      • Achsen dürfen NICHT aufeinander liegen
  • Kanonische Koordinatensystem
    Achsen stehen senkrecht aufeinander
  • Meta Koordinatensystem
    Die Achsen und der Ursprung eines Koordinatensystems muss immer bezüglich einer externen Referenz beschrieben werden (z.B. eine Ecke der Tafel).

Vektoren

  • Definition
    • Ein Vektor ist ein Tupel `(x_0, x_1,..., x_n)` mit `n` (float) Einträgen
    • `n`: Dimension des Vektors und des Koordinatensystem
  • Vektor beschreiben Veränderung, sie erhalten:
    • Eine Richtung
    • Eine Länge
  • Eigenschaften
    • Länge (Magnitude): `abs(vecv)=sqrt(x^2+y^2...)`
    • Richtung: `abs(vec(v_r))=vecv/abs(vecv)`
      → `vec(v_r)` ist jetzt normalisiert und hat die Länge 1
    • Multiplikation: Dot product (Punktprodukt)
      • `veca@vecb=(a_x*b_x+a_y*b_y+...)`
        → Es gilt:
        • kommutativ: `veca@vecb=vecb@veca`
        • distributiv: `veca(vecb+vecc)=veca@vecb+veca@vecc`
      • `veca@vecb=abs(a)*abs(b)*cos(theta)`
        → Für Einheitsvektoren:
        • `veca@vecb=1` `=>veca` und `vecb` haben dieselbe Richtung
        • `veca@vecb=1` `=>veca` und `vecb` sind entgegengesetzt
        • `veca@vecb=1` `=>veca` und `vecb` stehen senkrecht zueinander

Vektoren und Punkte

  • Ein Punkt beschreibt eine Position
  • Ein Vektor beschreibt ein Veränderung/Differenz
  • Operationen (`vecv`: Vektor, `p`: Punkt):
    • `vecv+vecv=vecv`
    • `vecv-vecv=vecv`
    • `p+p=nonsense!` (No visual interpretation)
    • `p-p=vecv` (difference vector)
    • `p+vecv=p` (dito)
    • `p-vecv=p` (translate)

Transformationen

  • Translation
    • Formular:
      `T(T_x,T_y)=[[T_x],[T_y]]`
      z.B: `[[-20],[500]]` zum Verschieben nach links um -20 und nach oben um 500
  • Skalierung
    • Skalierungsmatrix:
      • quadratische Matrix `(nxxn)`
      • Skalierungswerte nur auf der Hauptdiagonale
    • Formular:
      `S(S_x,S_y) = [[S_x,0],[0,S_y]]`
  • Rotation
    • Formular:
      `R(theta)=[[cos(theta),sin(theta)],[-sin(theta),cos(theta)]]`
      z.B: `R(pi/2)=[[cos(pi/2),sin(pi/2)],[-sin(pi/2),cos(pi/2)]]=[[0,1],[-1,0]]` zum Drehen um `90^@`

WebGL Code

  • WebGL Shader
    Im (Vertex) Shader ist Uniform Variablen zum Beschreiben der Transformationen (in GLSL-Datei).
    uniform vec3 a_translation;
    uniform matrix3 a_rotation;
    uniform matrix3 a_scale;
  • WebGL Client
    (Transformation-) Variable in Client-Code (z.B. in Script-Tag einer HTML-Seite) zuweisen
    var translationLocation = gl.getUniformLocation(shaderprogram, "a_translation");
    gl.useProgram(shaderprogram);
    gl.uniform3f(translationLocation, x, y, z);

Datentyp - Punkte

  • Definition
    Punkte definieren eine Position im Raum
  • Speichergröße
    • Bytes:
      • 1 Bit = true(1) / false(0) → kleinste Einheit
      • 1 Byte = 8 Bit = 256 Zustände → kleinste addressiertbare Einheit
    • Integer:
      • short = 16 Bit = 2 Bytes = 0...65K
      • int = 32 Bit = 4 Bytes = 4 Milliarden
      • long = 64 Bit = 8 Bytes = enorm...
    • Floating Point
      • float = 4 Bytes ≈ 7 Dezimalstellen
      • double = 8 Bytes ≈ 17 Dezimalstellen

Datentyp - Linien (Segmente)

  • Definition
    • 2 Varianten:
      • Start- und Endpunkt (zu speichern)
      • Startpunkt und Vektor zum Endpunkt (zu speichern)
    • Datengröße: 2 * 3 * 4 Bytes
      (Ein 3D-Punkt + 3D-Vektor oder 2 * 3D-Punkte)
  • Polylinien
    • Definition:
      • sind die Verkettung von Liniensegmente
      • `S_0=text(Startpunkt)`
      • `v_1, v_2, v_3, v_4, v_5,...=text(Punkte)`
        • → (n+1) Punkte für n Segmente
    • Datengröße: (n + 1) * 3 * 4 Bytes
  • Optimale Speicherung
    möglicherweise als Array von Liniensegmente speichern
    → Aber:
    • `v1=>v_(n-1)` muss doppelt gespeichert werden → mehr Speicherbedarf
    • Fehler, wenn `v_1` != `v_2`
    • Extra Code nötig um dies zu vermeiden
  • Polygon
    • Definition:
      • ist eine abgeschlossene Polylinie
      • n Punkte für n Segmente
    • Datengröße: n * 3 * 4 Bytes
    • Einfaches Polygon:
      • hat keine kreuzenden Kanten
      • kann immer durch Umsortierung der Punktenmenge erzeugt werden
    • Klassen:
      • konvex
        • beste Qualität (erwünscht)
        • jede Verbindung im Polygon
      • stern-förmig (star-convex)
        • mindestens einen Punkt, so dass jede Verbindung von diesem Punkt im Polygon
      • konkav
        • unerwünscht
        • existieren 2 Punkte im Polygon mit Verbindung nicht vollständig im Polygon

Datentyp - Dreiecke

  • Definition
    • Das Dreieck definiert sich über 3 Punkte
    • 3 Punkte dürfen nicht auf einer Geraden liegen
    • genau eine Ebene (das Dreieck immer flach)

  • Netztypen

Mesh data

  • 2 Listen
    • vorkommenden Punkten (unsortiert)
      • Points = [P1, P2, P3, ... Pv]
      • Speicherbedarf: v * 3 * 4 Bytes
    • Indices in die Punktliste, jeweils 3 Dreieck
      • indices = [... ca. 6 * v Elemente ...]
      • Speicherbedarf: ca. 6 * v * 4 Bytes
      • → Insgesamt: (v * 3 * 4) + (6 * v * 4) = 9 * v * 4 Bytes
    • Beispiel:
      • Ein Mesh hat:
        • 50.000 Punkte → Speicherbedarf: 50K * 3 * 4 Bytes
          with 3 is x, y, z of a vertex and 4 Bytes is Floating Point
        • 100.000 Dreiecke → Speicherbedarf: 100K * 3 * 4 Bytes
          with 3 is indices of a triangle and 4 Bytes ist (Floating Point)
      • Naive List von Dreiecken: 100K * 3 * 3* 4 Bytes = 10.8 MB
      • Effizienter Punkt/Index:
        (50K * 3 * 4 Bytes) + (100K * 3 * 4 Bytes) = 1.8 MB
  • Speicher optimieren
    Speichertyp für die Indices optimieren:
    • Bytes: 1 Byte genug für max. 256 Punkte
    • Integer: 2 Bytes (short) genug für max. 65K Punkte
    • Floating Point: 4 Bytes (float) genug für ca. 4 Milliarden Punkte
    → Wenn ein Mesh nicht mehr als 65K Punkte hat, es ist möglich 2 Bytes (short) anstatt 4 Bytes (float) zu nutzen.
    z.B. (50K * 3 * 4 Bytes) + (100K * 3 * 4 Bytes 2 Bytes) ≈ 1.14 MB
  • Dreiecksnormale
    Dreiecksnormale = das Kreuzprodukt der `ubrace((v_1-v_0))_(veca)` und `ubrace((v_2-v_1))_(vecb)` Vektoren
    `vecaxxvecb=[[a_1],[a_2],[a_3]]xx[[b_1],[b_2],[b_3]]=[[a_2*b_3-a_3*b_2],[a_3*b_1-a_1*b_3],[a_1*b_2-a_2*b_1]]`
  • Einfaches Polygon erstellen
    Two peasants - Algorithmus:
    1. Finde maximale und minimale x-Position
    2. Bilde eine gedachte Linie zwischen den Beiden
    3. Bilde 2 Arrays:
      • Alle Punkte unter der Linie
      • Alle Punkte über der Linie
    4. Sortiere das untere Array nach x-Koordinate
    5. Sortiere das obere Array nach umgekehrter x-Koordinate
    6. Bilde das einfache Polygon: [Start + untere Liste + Ende + obere Liste]

3D Datenformate

  • Datenformate
    • FBX: gebräuchlichste Format von Autodesk, ist Standard in Game-Development
    • PLY: Sehr einfach, sehr beschränkt
    • OBJ: Sehr einfach, etwas flexibler
    • DAE:
      • Von dem Khronos-Konsortium
      • COLLADA
      • XML-basiert
      • kann ganze Szenen mit Animation und Physik speichern.
    • glTF:
      • auch Khronos-Konsortium
      • JSON + binary
      • XML-basiert
      • ganze Szenen + Animation
  • glTF
    • Definition:
      • ist ein Dateiformat, dass eine komplette Szene mit animationen in einem JSON-Datei abspeichern kann
      • ist populär
      • von Facebook das Über-Format FBX zu glTF konvertiert und lässt sich direkt in Facebook einbinden
      • JSON Format direkt in Javascript/HTML einlesbar
    • Aufbau:
      Scene:
      • Index of Scene in Scenes
      Scenes:
      • contains different scenes, each with nodes[]
      • nodes[] contains indices of nodes
      Nodes:
      • contains different nodes, each with children[], mesh, transformations,...
      • children[] contains indices of nodes in here
      • mesh contains index of mesh in meshes
      buffers:
      • contains location of binary data and byte length
      bufferViews
      • contains index of buffer and allocation of binary data
      Binary data
      • a extra bin file goes with this glTF file
      • contains attributes in bin file
      images
      • contains textures as png, jpg,... files
      And other parts...
